# Collapsarr database migrations (Alembic)

Alembic owns the SQLite schema for Collapsarr. This directory ships **inside**
the installed package (`collapsarr/migrations/`) so `env.py`,
`script.py.mako`, and every revision under `versions/` are present in the built
wheel and the Docker image — not only in a source checkout.

**SQLite is the only supported and tested backend.** The environment enables
`render_as_batch=True` (SQLite's `ALTER TABLE` can't drop/alter columns
in place), plus `compare_type=True` and `compare_server_default=True` so
autogenerate notices column type / server-default drift against
`Base.metadata`.

## Runtime vs. authoring

- **Runtime** (the app, Docker, an installed wheel): the Alembic `Config` is
  built *in code* by `collapsarr.migrations.build_alembic_config(settings)` —
  `script_location` points at this packaged directory and `sqlalchemy.url`
  comes from `Settings.sqlalchemy_url`. **No `alembic.ini` is read at runtime.**
- **Authoring** (a repo checkout only): the repo-root `alembic.ini` provides
  the `alembic` CLI with `script_location`, the date-prefixed `file_template`,
  and a throwaway scratch DB (`./.alembic-scratch.db`, gitignored). It is used
  *solely* for `alembic revision --autogenerate` and `alembic history`.

> Boot-path note: as of COL-57 the production boot path is unchanged — the app
> still creates its schema via `init_db` (`Base.metadata.create_all` +
> `ensure_schema`). Wiring `upgrade head` into boot and switching fixtures over
> to migrations is the next slice (COL-58).

## Authoring workflow

Run from the **repo root** (the `alembic` CLI picks up `alembic.ini` there).
Requires the dev environment installed (`pip install -e ".[dev]"`).

1. Change or add SQLAlchemy models under `collapsarr/**/models.py`.
2. Autogenerate a revision against a scratch DB migrated to the current head:

   ```bash
   rm -f .alembic-scratch.db
   alembic upgrade head                       # bring scratch DB to current head
   alembic revision --autogenerate -m "short slug describing the change"
   ```

3. **Review the generated file** in `versions/`. Autogenerate is a starting
   point, not gospel — check the `upgrade()`/`downgrade()` bodies, especially
   for SQLite batch operations, data migrations, and server defaults.
4. Sanity-check it applies cleanly:

   ```bash
   rm -f .alembic-scratch.db && alembic upgrade head
   ```

5. Commit the new `versions/…py` file. Its filename is date-prefixed
   (`YYYY_MM_DD_<rev>_<slug>.py`); revision *identity* remains the Alembic hash
   + `down_revision` chain, so ordering is by the chain, not the date.

## Downgrades

Autogenerated `downgrade()` bodies are kept for local/dev convenience.
**Production is forward-only** — downgrades are not run against real installs.
